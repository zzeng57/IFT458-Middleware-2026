What I Learned
Cloning a Repository I learned that git clone creates a full local copy of a remote repository on my machine. Before cloning, I had to make sure my GitHub credentials were configured correctly using: bashgit config --global user.name "Name of the user" git config --global user.email "Useremail" Then I cloned the repo using: bashgit clone https://github.com/DineshSthapit/IFT458-Middleware-2026.git cd IFT458-Middleware-2026 Key takeaway: Always verify your credentials and repo URL before cloning, especially for private repositories where authentication matters.

Understanding Project Structure Opening the README.md first taught me the importance of reading documentation before writing any code. The project had two separate parts — a backend and a mobile frontend — each needing their own VS Code instance and terminal. This mirrors how real-world full-stack projects are structured.

Running the Backend (Node.js) I learned that environment variables (stored in config.env) are kept separate from the codebase for security reasons — things like database passwords and API keys should never be committed to GitHub. Starting the server with: bashnode server.js ...and then visiting http://localhost:4000 taught me how a backend server listens on a port and serves responses to clients.

Running the Mobile App (Expo / React Native) Using npx expo start and pressing w to open the web preview showed me how React Native can target multiple platforms (iOS, Android, Web) from a single codebase. I learned that Expo handles a lot of the build complexity behind the scenes.

The Git Workflow (Status → Add → Commit → Push) This was the most hands-on learning for me. I now understand each step: CommandWhat it doesgit statusShows which files are new, modified, or untrackedgit add .Stages all changes for the next commitgit commit -m "message"Saves a snapshot of staged changes with a descriptiongit pushUploads commits to the remote GitHub repository I repeated this cycle for each deliverable (ServerStudent.MD, screenshots, MobileStudent.MD), which reinforced that Git is a habit, not a one-time action.

Adding a Collaborator I learned how to give another GitHub user access to my repository by going to: Settings → Collaborators → Add People and entering dsthapit@asu.edu. This is how teams share and review each other's code in professional environments.

Overall Reflection This module connected everything — version control, backend servers, mobile frontends, and collaboration tools — into one real workflow. The biggest challenge was managing credentials and making sure the right GitHub account was authenticated. Going forward, I feel confident cloning repos, running full-stack projects locally, and pushing my work back to GitHub.
